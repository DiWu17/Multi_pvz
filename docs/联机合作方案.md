# PVZ 多人合作联机方案

## 一、模式概述

- **玩家人数**：2-4 人合作
- **核心玩法**：所有玩家共享同一片草坪，共用阳光池，协作种植植物抵御僵尸波次
- **区域规则**：无分区限制，所有人均可在草坪任意位置种植/铲除
- **难度缩放**：阳光掉落和僵尸数量随玩家人数动态增长

---

## 二、网络架构

### Host 模式（无独立服务器）

```
┌─────────────┐     ENet      ┌─────────────┐
│  Host(P1)   │◄────────────►│  Client(P2)  │
│  权威服务器  │               │              │
│  + 本地玩家  │     ENet      ├─────────────┤
│             │◄────────────►│  Client(P3)  │
│             │     ENet      ├─────────────┤
│             │◄────────────►│  Client(P4)  │
└─────────────┘               └─────────────┘
```

- 使用 Godot 4.x 内置 `ENetMultiplayerPeer`
- P1 作为 Host，同时承担服务器和玩家角色
- 所有游戏逻辑判定在 Host 端执行（权威服务器模型）
- 客户端发送操作请求，Host 验证后广播结果
- 后期可接入 Steam P2P 或 WebRTC 实现公网穿透

## 三、核心规则

### 3.1 共享阳光池

- 所有玩家共用一个 `sun_value`，Host 端维护唯一值
- 阳光掉落（天空/向日葵产出）所有玩家均可点击收集
- 点击判定：谁先点到算谁收（Host 端按到达时间排序，防重复收取）
- 种植扣除阳光时，Host 验证余额后再执行

### 3.2 草坪完全共享

- 所有玩家可在任意空闲格子种植
- 所有玩家可铲除任意植物（包括他人种植的）
- 植物记录 `owner_peer_id` 属性，用于 UI 显示归属（仅展示，不限制操作）
- 每个玩家有独立的卡组选择和卡片冷却

### 3.3 难度缩放公式

| 项目 | 公式 | 示例（2人/3人/4人） |
|------|------|---------------------|
| 僵尸数量倍率 | `1.0 + (player_count - 1) * 0.5` | 1.5x / 2.0x / 2.5x |
| 阳光掉落倍率 | `1.0 + (player_count - 1) * 0.35` | 1.35x / 1.7x / 2.05x |
| 阳光掉落间隔 | 原始间隔不变，增加同时存在的阳光上限 | — |
| 僵尸波次间隔 | 不变 | — |
| 起始阳光 | `start_sun * player_count * 0.75` | 75/112/150（基于50） |

> 设计意图：僵尸增长略快于阳光增长，确保人多时仍有压力，需要更好的协作。

### 4.2 需要改造的现有文件

#### `main_game_manager.gd` — 改动程度：大

- 增加 `is_multiplayer` 标志
- `_ready()` 中根据 Host/Client 决定是否启动本地逻辑
- 胜利/失败判定只在 Host 端执行，结果通过 RPC 广播
- 游戏进度状态 `game_progress` 由 Host 同步给所有客户端
- 难度缩放：在初始化时根据 `player_count` 修改 `game_para`

```gdscript
# main_game_manager.gd 新增
var is_multiplayer: bool = false
var player_count: int = 1

func _apply_multiplayer_scaling():
    if not is_multiplayer: return
    var zombie_scale = 1.0 + (player_count - 1) * 0.5
    var sun_scale = 1.0 + (player_count - 1) * 0.35
    game_para.zombie_multy *= zombie_scale
    game_para.start_sun = int(game_para.start_sun * player_count * 0.75)
    # sun_scale 传给 DaySunsManager
```

#### `hand_manager.gd` — 改动程度：大

- 每个客户端保留自己的 HandManager 状态机（本地 UI 交互）
- 种植/铲除操作不直接执行，改为发送 RPC 请求

```gdscript
# 客户端请求种植
func _on_click_cell_multiplayer(row: int, col: int):
    var plant_type = current_selected_plant_type
    network_manager.request_plant.rpc_id(1, plant_type, row, col)

# Host 端验证并广播
@rpc("any_peer", "call_local", "reliable")
func request_plant(plant_type: int, row: int, col: int):
    if not multiplayer.is_server(): return
    var peer_id = multiplayer.get_remote_sender_id()
    if _validate_plant(peer_id, plant_type, row, col):
        _execute_plant.rpc(plant_type, row, col, peer_id)
```

#### `card_manager.gd` — 改动程度：中

#### `day_suns_manager.gd` — 改动程度：中

- 阳光生成只在 Host 端执行
- 生成后通过 RPC 广播位置，客户端创建本地阳光实例
- 点击收集：客户端发请求 → Host 判定（防重复）→ 广播消失 + 加阳光

```gdscript
# Host 端生成阳光
func _spawn_sun():
    if not multiplayer.is_server(): return
    var pos = Vector2(randf_range(50, 750), -40)
    var target_y = randf_range(130, 550)
    var sun_id = _next_sun_id()
    _broadcast_sun_spawn.rpc(sun_id, pos, target_y)

# 客户端请求收集
@rpc("any_peer", "reliable")
func request_collect_sun(sun_id: int):
    if not multiplayer.is_server(): return
    if sun_id in active_suns:
        active_suns.erase(sun_id)
        var sun_scale = 1.0 + (player_count - 1) * 0.35
        var value = int(25 * sun_scale)  # 缩放后的阳光值
        _broadcast_sun_collected.rpc(sun_id, value)
```

## 五、同步策略

| 数据类型 | 同步方式 | 可靠性 | 频率 |
|----------|----------|--------|------|
| 僵尸位置/动画 | MultiplayerSynchronizer | unreliable | 每帧（插值） |
| 僵尸生成/死亡 | RPC | reliable | 事件触发 |
| 植物种植/铲除 | RPC | reliable | 事件触发 |
| 阳光值变化 | RPC | reliable | 事件触发 |
| 阳光掉落位置 | RPC | reliable | 事件触发 |
| 阳光收集 | RPC | reliable | 事件触发 |
| 子弹生成/命中 | Host 本地计算 + 效果 RPC | reliable | 事件触发 |
| 玩家光标位置 | RPC | unreliable | 10Hz |
| 波次进度 | RPC | reliable | 事件触发 |
| 游戏状态切换 | RPC | reliable | 事件触发 |
| 卡片冷却 | 各客户端本地计算 | — | 本地 |

## 六、多人交互细节

### 6.1 实时光标共享（类似杀戮尖塔合作模式）

每个玩家的光标、选中状态对其他所有人实时可见，提升协作感知：

**光标外观**
- 每个玩家有专属颜色的光标（P1红/P2蓝/P3绿/P4黄）
- 光标旁显示玩家昵称标签（小字，半透明）

**选中状态同步**
- **选中植物时**：其他玩家能看到该光标旁带有对应植物的半透明缩略图
- **选中铲子时**：光标变为对应颜色的铲子图标
- **悬停格子时**：该格子显示对应颜色的半透明高亮边框
- **空闲状态**：仅显示颜色光标箭头

**同步数据结构**
```gdscript
# 每 100ms (10Hz) 通过 unreliable RPC 广播
class CursorState:
    var position: Vector2        # 光标屏幕坐标
    var hovered_cell: Vector2i   # 当前悬停的格子 (row, col)，无则 (-1,-1)
    var held_type: String        # "none" / "plant" / "shovel"
    var held_plant_type: int     # 选中的植物类型枚举值（held_type == "plant" 时有效）

@rpc("any_peer", "unreliable")
func sync_cursor(peer_id: int, state: Dictionary):
    _update_remote_cursor(peer_id, state)
```

**性能与体验**
- 远端光标使用插值平滑移动，避免 10Hz 下的跳跃感
- 可在设置中关闭/降低他人光标显示（关闭 / 仅显示格子高亮 / 全部显示）
- 选卡阶段不显示他人光标，仅游戏阶段显示

### 6.2 种植冲突处理

- 两人同时点同一格子：Host 按 RPC 到达顺序，先到先得
- 失败方客户端收到拒绝响应，播放"无法种植"提示音
- 不扣除失败方阳光

### 6.3 卡组选择

- 每个玩家独立选卡（各自的 CardSlot UI）
- 选卡阶段所有人准备完毕后 Host 启动游戏
- 不同玩家可选相同植物，冷却独立计算

### 6.4 游戏流程

```
1. Host 创建房间 → 其他玩家加入
2. 大厅界面：显示玩家列表、准备状态
3. 全员准备 → Host 点击开始
4. 进入选卡界面（每人独立选卡）
5. 全员选完 → 进入游戏
6. 共享草坪、共享阳光、合作种植
7. 通关/失败 → 结算界面 → 返回大厅或下一关
```

## 七、断线处理

| 场景 | 处理方式 |
|------|----------|
| 客户端断线 | 其植物保留在场上，卡片冷却冻结，位置空出 |
| 客户端重连 | 同步当前游戏状态（阳光、格子、僵尸、波次） |
| Host 断线 | 游戏结束，所有客户端弹出提示返回主菜单 |
| 选卡阶段断线 | 移除该玩家，重新计算难度缩放 |
